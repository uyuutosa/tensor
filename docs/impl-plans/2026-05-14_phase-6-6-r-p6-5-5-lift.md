## Milestones

The order is dependency-driven; each milestone exits when its tests pass on Linux + macOS (the WebGPU + Windows-WebGPU scoping from Phase 6.5 R-P6.5.4 is unchanged).

### P6.6.M1 — Types-owner module split

**Goal**: extract the shared C++ type registrations from `python/src/_tensor_native.cpp` into a new `python/src/_tensor_types.cpp` that builds into `_tensor_types.so`, importable on its own.

**Deliverables**:

- `python/src/_tensor_types.cpp` (new) — owns:
  - `nb::class_<Axis>` registration.
  - `nb::class_<DynamicShape>` registration.
  - `nb::class_<DynamicTensor<double>>` + `DynamicTensor<float>` registrations.
  - `nb::class_<ag::ScalarVariable<double>>` + `ScalarVariable<float>` registrations.
  - `nb::class_<ag::Tape>` registration.
  - `nb::class_<tex::Expression>` registration.
  - `nb::class_<tex::Evaluator<double>>` + `Evaluator<float>` registrations.
  - The `from_numpy_typed` / `to_numpy_typed` helpers exposed as module-level free functions on `_tensor_types` so the kernel modules can call them without re-defining.
- `python/CMakeLists.txt` — new `_tensor_types` extension module target (always built; not backend-conditional).
- `python/tensor/__init__.py` — imports `_tensor_types` first (raises `RuntimeError: tensor type-owner module missing; reinstall with pip install --force-reinstall tensor-named-axis` if absent), then proceeds to backend discovery.
- `python/tests/test_types_owner.py` (new) — `import tensor._tensor_types` works; `_tensor_types.Axis("i", 4)` constructs; `_tensor_types.from_numpy(np.zeros((3, 4)), ["i", "j"])` returns a `DynamicTensor`.

**Exit criteria**: `pip install -e .` produces both `_tensor_types.so` and the existing reference `_tensor_native_reference.so`; `python -c "import tensor; print(tensor.current_backend())"` still works.

**Effort**: M (1 day).

### P6.6.M2 — Kernel modules import-and-use

**Goal**: each `_tensor_native_<backend>.cpp` imports `_tensor_types` and registers only its own kernel-call entry points, sharing the type universe.

**Deliverables**:

- `python/src/_tensor_native.cpp` (existing, renamed `_tensor_native_kernel.cpp` to clarify the split) — gutted of `nb::class_<...>` registrations; retains only:
  - `nb::module_::import_("tensor._tensor_types")` at the top of `NB_MODULE`.
  - `m.def("contract", ...)` × 2 (double, float).
  - `m.def("hello", ...)` placeholder kept for backend identity.
  - `autograd` submodule with `m.def("backward", ...)` and the `SGD.step` function — `ScalarVariable` itself is registered in `_tensor_types`.
  - `tex` submodule with `m.def("parse", ...)` — `Expression` and `Evaluator` are registered in `_tensor_types`.
- `m.attr("__backend__")` stays in each kernel module (so the loader can tell which one is which).
- `python/extras/tensor-named-axis-eigen/pyproject.toml` — runtime dep added: `tensor-named-axis-types==<same-version>` (the type-owner module is its own distribution; see ADR-0020 §Follow-ups M2 investigation).
- Same change for `tensor-named-axis-webgpu/pyproject.toml`.
- `python/extras/tensor-named-axis-types/pyproject.toml` (new) — packages only `_tensor_types.so` + the `tensor/` PEP-420 namespace stub.

**Exit criteria**: `pip install -e .[eigen]` brings in both `tensor-named-axis-types` and `tensor-named-axis-eigen`; `python -c "import tensor; tensor.set_backend('eigen'); print(tensor.contract(...))"` works; reference and Eigen kernels share the same `DynamicTensor` Python class (verified via `tensor.from_numpy(...).__class__ is tensor._tensor_types.DynamicTensor`).

**Effort**: L (2 days).

### P6.6.M3 — `set_backend()` true in-process rebind

**Goal**: `set_backend("reference")` → `contract(...)` → `set_backend("eigen")` → `contract(...)` works in one process.

**Deliverables**:

- `python/tensor/__init__.py` — `set_backend(name)` now actually swaps the module-level bindings:
  - Removes the R-P6.5.5 raising path for cross-backend switches.
  - Imports the requested `_tensor_native_<name>` (if not already imported) and rebinds `tensor.contract`, `tensor.autograd.backward`, `tensor.tex.parse` to that module's exports.
  - Updates `tensor._active_backend` state.
  - Backward-compat note: if the *types* module is not present, raise the M1 error.
- `python/tests/test_set_backend.py` — new case `test_in_process_switch_between_installed_backends`:
  - `set_backend("reference")`, contract a 4×7 × 7×5, capture result A.
  - `set_backend("eigen")`, contract the same shapes, capture result B.
  - Assert `max(abs(A - B)) < 1e-12` (double) / `1e-5` (float).
  - Skip the case if Eigen extra is not installed.
- `python/notebooks/05_swappable-backends.ipynb` §6 — "Known limitation — R-P6.5.5" section rewrites to "Historical — resolved by Phase 6.6 / ADR-0020"; the demo now shows a live `set_backend("reference")` → `set_backend("eigen")` comparison with timing.

**Exit**: in-process switch test passes; the swappable-backends notebook renders the comparison.

**Effort**: M (1 day).

## Out of scope

- **Windows WebGPU wheel** — still R-P6.5.4; Phase 6.6 does not touch that scoping.
- **conda-forge feedstock** — stays single-output (reference only). Add the type-owner + extras outputs as a Phase 7+ follow-up.
- **Per-thread backend selection**. `set_backend()` remains process-global; threading concurrency is undefined. Document via a one-line note in the how-to; do not implement thread-local backend state.
- **`set_backend()` mid-tensor-lifetime** — switching backends does not migrate existing `DynamicTensor` instances. Newly-constructed tensors route through the new backend; existing tensors retain their original kernel binding. Document via a one-line note.

The Phase 6.5 cibuildwheel matrix (~52 wheels per release) grows by one extension target × 4 OS/arch × 5 CPython = 20 wheels for `tensor-named-axis-types`. Total: ~72 wheels per release.

## Risks

- **R-P6.6.1 — ABI co-build skew.** If the types-owner module and kernel modules are ever built against different `tensor::core` headers or different nanobind ABI versions, the `nb::module_::import_` rebind silently mismatches types. Mitigation: cibuildwheel matrix builds all four `.so` files in the same job per slot; `python/tests/test_types_owner.py` adds a `__abi_signature__` cross-check to detect skew at import time.
- **R-P6.6.2 — Multi-process `import` race.** If two Python processes hit a partially-installed wheel set (types module installed, kernel module not yet), the second process's `import tensor` raises. Mitigation: order extras dependency `tensor-named-axis-eigen` → depends on → `tensor-named-axis-types`, so `pip` installs the types module first.
- **R-P6.6.3 — Type registration order on `set_backend()`.** If kernel module B is imported before kernel module A but A's types are needed by B, nanobind raises. Mitigation: type-owner module imports both happen before any kernel module's `m.def()` body runs (per nanobind 2.x's lazy-attribute semantics) — verified in M1 smoke test.
